# Чек-лист для полноценной тестовой версии

Что нужно исправить и проверить на всех уровнях перед выкладкой тестовой (staging/preview) версии.

---

## Решение (что сделать по шагам)

1. **Окружение** — в корне проекта скопировать `.env.example` в `.env.local` и подставить реальные значения:
   - `EXPO_PUBLIC_SUPABASE_URL`, `EXPO_PUBLIC_SUPABASE_ANON_KEY` (из Supabase)
   - `EXPO_PUBLIC_BACKEND_URL` / `EXPO_PUBLIC_API_URL` (URL тестового backend)
   - `EXPO_PUBLIC_PROJECT_ID` (Expo), `EXPO_PUBLIC_ENV=staging`
2. **Backend** — в `backend/.env` задать те же Supabase, `JWT_SECRET`, `ALLOWED_ORIGINS` (с URL тестового фронта).
3. **Ресурсы** — выполнить `./scripts/prepare-production-build.sh`, чтобы появились `local/assets/notification_icon.png` и `notification_sound.wav`.
4. **Проверка** — запустить приложение, проверить вход, 4 таба (Чаты, Контакты, Настройки, Профиль) и разделы из Профиля; убедиться, что API отвечает (нет CORS/404).

Остальное — по детальному чек-листу ниже.

## 1. Окружение и конфигурация

### 1.1 Переменные окружения (Frontend — `.env` / `.env.local`)

| Переменная | Проблема | Действие |
|------------|----------|----------|
| `EXPO_PUBLIC_SUPABASE_URL` | В `.env` указан `https://placeholder.supabase.co` | Заменить на реальный URL проекта Supabase для тестового окружения |
| `EXPO_PUBLIC_SUPABASE_ANON_KEY` | Указан `placeholder-anon-key-for-local-dev` | Подставить реальный anon key из Supabase |
| `EXPO_PUBLIC_BACKEND_URL` / `EXPO_PUBLIC_API_URL` | Должны указывать на тестовый backend | Задать URL тестового API (например Vercel preview или отдельный инстанс) |
| `EXPO_PUBLIC_RORK_API_BASE_URL` | Используется в tRPC | Проверить, что совпадает с тестовым backend |
| `EXPO_PUBLIC_PROJECT_ID` | Для Expo Push | Задать реальный project ID из Expo |
| `EXPO_PUBLIC_ENV` | Для логирования/аналитики | Задать `stagтing` или `test` для тестовой версии |
| `EXPO_PUBLIC_ENABLE_SENTRY` | Опционально | `true` для тестовой версии, чтобы ловить ошибки |
| `EXPO_PUBLIC_DEMO_AUTO_LOGIN` | Только для веб-демо | В тестовой сборке лучше не использовать или выключить |

### 1.2 Переменные окружения (Backend — `backend/.env`)

| Переменная | Действие |
|------------|----------|
| `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` | Реальные ключи тестового Supabase |
| `DATABASE_URL` | Подключение к тестовой БД (если используется отдельно от Supabase) |
| `JWT_SECRET` | Уникальный секрет для тестового окружения (не от прод) |
| `OPENAI_API_KEY` | Ключ для AI-функций (если нужны в тесте) |
| `ALLOWED_ORIGINS` | Добавить URL тестового фронта (Expo web, tunnel, preview) |
| `NODE_ENV` | `staging` или `development` по вашему процессу |

### 1.3 EAS / Сборка

| Файл / место | Проблема | Действие |
|--------------|----------|----------|
| `eas.json` → `submit.production.ios` | `appleId: "your-apple-id@example.com"` и плейсхолдеры | Для тестовой версии можно не трогать; для сабмита в Store подставить реальные Apple ID, Team ID, ASC App ID |
| `eas.json` → `submit.production.android` | `serviceAccountKeyPath` | Указать путь к ключу сервисного аккаунта для Google Play |
| `app.json` | `version`, `ios.buildNumber`, `android.versionCode` | Для тестовой сборки поднять версию/билд, чтобы отличать от предыдущих установок |

## 2. API и Backend

### 2.1 CORS и доступность

- В `backend/index.ts` для теста разрешены localhost и 127.0.0.1 — для тестового деплоя убедиться, что в `ALLOWED_ORIGINS` добавлены:
  - URL тестового веб-клиента (например `https://xxx.rork.live` или preview URL),
  - при необходимости `exp://` для Expo Go / dev client.
- После деплоя backend проверить вручную: открыть в браузере `https://<backend-url>/api/trpc` (или ваш health/trpc endpoint) и убедиться, что запросы с тестового фронта не режутся CORS.

### 2.2 Тестовые скрипты backend

В `backend/test-*.ts` захардкожен URL `https://backend-three-mauve-67.vercel.app`:

- Либо заменить на переменную окружения (например `BACKEND_URL`) и передавать тестовый URL при запуске скриптов,
- Либо явно документировать, что скрипты бьют в конкретный инстанс, и не запускать их против прод-окружения.

### 2.3 Supabase

- В `backend/trpc/routes/test/db-check.ts` и других местах проверяется наличие `SUPABASE_URL` и `SUPABASE_ANON_KEY`. Для тестовой версии они должны быть заданы и указывать на тестовый проект Supabase (не прод).

## 3. Безопасность и авторизация

### 3.1 Демо-режим

- В `_layout.tsx` условие `EXPO_PUBLIC_DEMO_AUTO_LOGIN === '1'` включает автологин только на веб и в dev. Для тестовой сборки решить: отключить переменную в staging или оставить только для внутренних тестов.

### 3.2 Токены и ключи

- Нигде в коде не должны остаться продовые ключи (Supabase, OpenAI, JWT_SECRET). Проверить, что в репозитории нет закоммиченных `.env` с реальными прод-значениями (используйте `.env.example` и секреты в CI/EAS).

### 3.3 Роли и доступ

- Для тестовой версии проверить сценарии: родитель, ребёнок, переход по экранам (Чаты, Контакты, Настройки, Профиль, разделы из Профиля). Убедиться, что ограничения по ролям работают так же, как задумано.

## 4. Ресурсы и ассеты

### 4.1 Уведомления (обязательно для нативных сборок)

| Ресурс | Где используется | Действие |
|--------|-------------------|----------|
| `local/assets/notification_icon.png` | `app.json` → expo-notifications | Должен существовать. При отсутствии — скрипт `scripts/prepare-production-build.sh` копирует из `assets/images/icon.png`; для тестовой сборки запустить подготовку или положить файл вручную |
| `local/assets/notification_sound.wav` | `app.json` → sounds | Должен существовать, иначе возможны ошибки при сборке/уведомлениях. Положить файл или отключить звуки в конфиге, если не нужны |

### 4.2 Прочие ассеты

- В `utils/customEmojis.ts` есть URL `https://example.com/kiku-logo.png` — заменить на реальный URL или локальный ассет, если этот эмодзи используется в тесте.
- Все пути к иконкам и сплешу в `app.json` (например `./assets/images/...`) проверить: файлы на месте и корректно открываются.

## 6. Мониторинг и ошибки

### 6.1 Sentry (если используется)

- В тестовой версии задать `EXPO_PUBLIC_ENABLE_SENTRY=true` и отдельный `EXPO_PUBLIC_ENV=staging` (или `test`), чтобы в Sentry можно было фильтровать по окружению и не смешивать с продом.
- В backend задать `SENTRY_ENVIRONMENT=staging` (или аналог) и при необходимости отдельный DSN для теста.

### 6.2 Обработка ошибок

- Error boundary в `_layout.tsx` уже показывает сообщение и кнопку «Попробовать снова» — убедиться, что в тестовой сборке при падении экрана пользователь видит понятный текст (при необходимости вывести `state.message` только в не-прод или подставить дружественный текст).

### 6.3 Сеть и tRPC

- В `lib/trpc.ts` при недоступности backend выводится предупреждение в консоль. Для тестовой версии убедиться, что `EXPO_PUBLIC_BACKEND_URL` / `EXPO_PUBLIC_API_URL` / `EXPO_PUBLIC_RORK_API_BASE_URL` указывают на один и тот же тестовый backend и что при отключённом backend пользователь видит адекватное сообщение (если такое есть в UI).

## 7. Тесты и качество

### 7.1 Запуск тестов

- Выполнить:  
  `bun run test:unit:coverage`  
  Убедиться, что все нужные тесты зелёные и что падение тестов не допускается в пайплайне тестовой сборки (если такой есть).

### 7.2 Критичные сценарии

- Вручную или E2E проверить: вход/выход, выбор роли, регистрация родителя/ребёнка, список чатов, открытие чата, переход в Настройки и Профиль, открытие разделов из Профиля (Звонки, Достижения, Уроки, Аналитика, Кастомные эмодзи, О приложении), настройки безопасности, push-уведомления (если включены).

## 8. Сборка и деплой тестовой версии

### 8.1 Подготовка

1. Заполнить/проверить переменные окружения для тестового окружения (п. 1.1, 1.2).
2. Запустить при необходимости `scripts/prepare-production-build.sh` (иконка и звук уведомлений).
3. Убедиться, что в EAS задан нужный профиль (например `preview` для внутренней тестовой сборки).

### 8.2 Сборка

- iOS (preview): `eas build --profile preview --platform ios`
- Android (preview): `eas build --profile preview --platform android`
- Web: проверить, что после `bun run start:web` (или аналога) приложение открывается и подключается к тестовому API.

### 8.3 После деплоя

- Проверить установку на реальных устройствах (iOS/Android) и в браузере.
- Проверить глубокие ссылки и переходы из уведомлений (если используются).
- Убедиться, что версия и билд в настройках/«О приложении» соответствуют тестовой сборке.
