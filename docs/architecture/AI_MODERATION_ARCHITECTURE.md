# Архитектура AI-модерации и создание похожей нейросети

**Дата:** 18 февраля 2026  
**Контекст:** KIKU — проактивная защита детей; документ описывает текущую архитектуру и как построить похожую систему (в т.ч. свою модель).

---

## 1. Текущая архитектура в KIKU (высокоуровнево)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           КЛИЕНТ (React Native / Expo)                        │
├─────────────────────────────────────────────────────────────────────────────┤
│  MonitoringContext / экран чата                                               │
│         │                                                                     │
│         ▼                                                                     │
│  AIModerationService.analyzeMessageWithAI(text)                                │
│         │                                                                     │
│         ├── 1. Быстрые правила (regex) ──► critical? → сразу return            │
│         │                                                                     │
│         ├── 2. Backend AI (если provider !== 'local')                          │
│         │         │                                                           │
│         │         └──► trpcClient.ai.analyzeMessage.query({ text })            │
│         │                      │                                              │
│         └── 3. Fallback: analyzeWithAdvancedHeuristics(text)                  │
│                                                                               │

┌─────────────────────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────────────────────┤
│  ai.analyzeMessage (rate-limited)                                             │
│         │                                                                     │
│         └──► OpenAI Moderation API (POST /v1/moderations)                      │
│                   │                                                           │
│                   └──► mapOpenAIToRiskLevel(flagged, category_scores)           │
│                        mapCategories(categories)                              │
│                        return { riskLevel, reasons, confidence, categories }   │
└─────────────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                     ВНЕШНИЕ СЕРВИСЫ (опционально)                             │

```

**Идея:** сначала быстрые и жёсткие правила (критичные угрозы), потом облачный AI через backend (ключ не светится на клиенте), при недоступности — улучшенные эвристики. Единый контракт — `RiskAnalysis` (riskLevel, confidence, reasons, categories).

## 2. Контракт данных (RiskAnalysis)

Во всём стеке используется один формат:

```ts
type RiskLevel = 'safe' | 'low' | 'medium' | 'high' | 'critical';

interface RiskAnalysis {
  riskLevel: RiskLevel;
  confidence: number;   // 0..1
  reasons: string[];
  categories: string[]; // hate, self_harm, sexual_minors, violence, ...
}
```

Любая «похожая нейросеть» или новый провайдер должны возвращать именно этот контракт (на бэкенде делаем маппинг из сырых ответов API/модели).

---

## 3. Два пути: API vs своя модель

### Вариант A: Использовать готовый API (как сейчас)

- **Плюсы:** быстро, не нужны данные и ML-инфраструктура, уже есть модерация под вредный контент.
- **Минусы:** зависимость от провайдера, стоимость по запросам, кастомизация ограничена.
- **Что нужно:** backend-proxy (как `ai.analyzeMessage`), env `OPENAI_API_KEY`, маппинг ответа в `RiskAnalysis`.

### Вариант B: Своя нейросеть (модерация / классификация риска)

- **Плюсы:** полный контроль, можно дообучать под домен (детская безопасность, локальный язык), нет платы за вызов.
- **Минусы:** нужны разметка, обучение, деплой, мониторинг качества.

Дальше — архитектура и шаги именно для **варианта B** (похожая нейросеть и её создание).

## 4. Архитектура «похожей нейросети» (своя модель)

### 4.1 Роль модели в пайплайне

Своя модель не заменяет весь пайплайн, а встаёт **вместо** вызова OpenAI (или параллельно для A/B).

```
Текст сообщения
      │
      ▼
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────────┐
│ Правила (regex)  │────►│ Критичный риск?  │────►│ Да → return critical  │
└─────────────────┘     └──────────────────┘     └─────────────────────┘
      │ Нет                           │ Нет
      ▼                              ▼
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────────┐
│ Ваша модель     │────►│ RiskLevel +      │────►│ combineAnalyses()   │
│ (текст → класс) │     │ confidence,      │     │ с правилами → return │
└─────────────────┘     │ categories       │     └─────────────────────┘
                         └──────────────────┘
```

То есть архитектура «похожей нейросети» — это: **вход (текст) → одна или несколько моделей → выход в формате RiskAnalysis**.

### 4.2 Выбор типа модели

| Подход | Плюсы | Минусы | Когда брать |
|--------|--------|--------|-------------|
| **Готовый Moderation API** (OpenAI и т.п.) | Быстро, качество | Стоимость, кастомизация | MVP, первый запуск |
| **Классификатор на BERT/RoBERTa** (fine-tune) | Хорошее качество, свой датасет | Нужны данные и GPU | Есть разметка 1–5k+ примеров |
| **Легкая модель (DistilBERT, small transformer)** | Быстрый инференс, дешевый деплой | Чуть хуже точность | Ограничения по латентности/ресурсам |
| **Правила + эвристики** | Ноль затрат, предсказуемо | Слабо против новых формулировок | Fallback или низкий бюджет |

Для «похожей нейросети» в духе KIKU разумный целевой вариант — **бинарный или мультиклассовый классификатор риска** (например, 5 классов: safe, low, medium, high, critical) на базе **предобученного текстового энкодера** (BERT/RoBERTa для русского + английского или multilingual).

### 4.3 Входы и выходы модели

- **Вход:** один текст сообщения (строка, можно ограничить длину, например 512 токенов).
- **Выход (сырой):**
  - класс риска (0–4 или 5 классов);
  - опционально: вероятности по классам (для `confidence`) и мультилейбл по категориям (bullying, grooming, threats, …).
- **Маппинг в RiskAnalysis:** класс → `riskLevel`, max вероятность → `confidence`, выбранные категории → `categories`, короткие подписи → `reasons`.

### 4.4 Где считать модель

- **Вариант 1 — отдельный сервис (рекомендуется):** микросервис на Python (FastAPI/Flask), держит модель в памяти, отдаёт JSON `{ "riskLevel", "confidence", "reasons", "categories" }`. Backend KIKU вызывает этот сервис вместо OpenAI (или как альтернативный провайдер).
- **Вариант 2 — внутри backend:** если позже будет Node-based рантайм для инференса (ONNX и т.п.) — можно вызывать модель из того же процесса, что и tRPC. Сейчас в KIKU бэк на JS/TS, поэтому проще вынести инференс в отдельный сервис.

Архитектура тогда такая:

```
Backend (Node)  ── HTTP ──►  Сервис модерации (Python)  ──►  Модель (PyTorch/ONNX)
                                    │
                                    └── возвращает RiskAnalysis (JSON)
```

## 5. Пошаговый план создания похожей нейросети

### Шаг 1: Сбор и разметка данных

- **Источники:** синтетика (шаблоны угроз, буллинга, груминга), публичные датасеты модерации (с соблюдением лицензий и этики), анонимизированные фразы из тестовых сценариев.
- **Разметка:** каждому примеру — `riskLevel` и при необходимости `categories`. Желательно хотя бы 1–2k примеров на класс для стабильного качества.
- **Валидация:** отложенная выборка и кросс-валидация по классам; отдельно замерить false positive (чтобы не помечать безопасное как риск).

### Шаг 2: Выбор базовой модели и обучение

- Взять предобученный энкодер (например, `rubert-base`, `xlm-roberta-base` для мультиязычности).
- Добавить голову классификации: один слой (или два) с выходом в 5 классов (и опционально мультилейбл по категориям).
- Обучить на своих данных (PyTorch или Transformers), сохранить чекпоинт и при необходимости экспортировать в ONNX для быстрого инференса.

### Шаг 3: Оценка и пороги

- На валидации подобрать пороги по вероятности для перевода в `riskLevel` (как сейчас в коде: 0.9 → critical, 0.7 → high, 0.5 → medium).
- Зафиксировать метрики: точность по классам, доля false positive, recall по critical/high.

### Шаг 4: Сервис инференса

- Написать небольшой API (FastAPI): вход — `{ "text": "..." }`, выход — тот же контракт, что и у OpenAI (или сразу `RiskAnalysis`).
- Загружать модель при старте, обрабатывать батчами при необходимости.
- Добавить лимиты и таймауты (как в rate limit для `ai.analyzeMessage`).

### Шаг 5: Интеграция в KIKU

- В backend добавить провайдер `custom` или `self-hosted`: если задан `MODERATION_SERVICE_URL`, вызывать его вместо OpenAI.
- Маппинг ответа сервиса в `RiskAnalysis` и возврат через тот же `ai.analyzeMessage` (или отдельная процедура с общим контрактом).
- Оставить правила и fallback как есть: сначала правила, при отсутствии критичного — вызов своей модели, при ошибке — эвристики.

### Шаг 6: Мониторинг и дообучение

- Логировать запросы и ответы (без персональных данных) для переразметки сомнительных случаев.
- Периодически дообучать модель на новых примерах и перевыпускать сервис.

## 6. Краткая сводка по файлам в KIKU

| Компонент | Файл | Назначение |
|-----------|------|------------|
| Правила + оркестрация AI | `constants/AIModerationService.ts` | Правила → backend AI или эвристики, combineAnalyses |
| Правила (чистые) | `utils/riskEvaluation.ts` | Детерминированная оценка по ключевым словам |
| Конфиг и вызов API | `utils/aiService.ts` | getAIConfig, кэш, вызов OpenAI (с ключом только на backend) |
| Backend proxy | `backend/trpc/routes/ai/analyze-message.ts` | Rate limit, вызов OpenAI Moderation, маппинг в RiskAnalysis |
| Типы | `constants/types.ts` | RiskLevel, RiskAnalysis |

Чтобы «подменить» нейросеть на свою — достаточно заменить вызов в `backend/trpc/routes/ai/analyze-message.ts` на запрос к вашему сервису и маппинг его ответа в тот же `RiskAnalysis`.

## 7. Что можно сделать дальше

- Добавить в репозиторий папку `ml/` или отдельный репозиторий с скриптами: разметка, обучение, экспорт в ONNX и Docker-образ сервиса модерации.
- Описать в отдельном документе формат API своего сервиса (request/response) и переменные окружения (`MODERATION_SERVICE_URL`, таймаут).
- В коде backend ввести провайдер `self-hosted` и вызывать свой сервис при наличии URL.

Если нужно, могу набросать конкретный пример API контракта для вашего сервиса модерации и фрагмент кода интеграции в `analyze-message.ts`.
